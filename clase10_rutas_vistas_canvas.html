Al finalizar vamos a tener la primera URL real del sistema de turnos respondiendo desde Django:

Conceptos · 20 min
¿Qué es una app en Django?
Un proyecto Django está compuesto de aplicaciones. Cada app agrupa una funcionalidad específica del sistema. Es la forma que tiene Django de organizar código a medida que el sistema crece.
Proyecto vs. Aplicación
La distinción es importante y se confunde seguido al principio:
Proyecto
sistema_turnos/
El contenedor global. Define la configuración, la base de datos, los secretos, las apps instaladas. Hay uno solo por sistema.
Ej: "El sistema de gestión de ANSES" como conjunto.
Aplicación
turnos/, usuarios/, reportes/
Un módulo que resuelve una función específica. Un proyecto puede tener varias apps. Cada una es reutilizable e independiente.
Ej: El módulo de turnos, el módulo de login, el módulo de reportes.
Analogía del Estado
Pensar en un ministerio. El Ministerio de Salud es el proyecto. Dentro tiene direcciones independientes: Dirección de Epidemiología, Dirección de Hospitales, Dirección de Medicamentos. Cada una tiene su propio personal, sus propias reglas, su propio presupuesto — pero todas forman parte del mismo ministerio.

En Django: el proyecto es el ministerio, las apps son las direcciones.
Crear la app turnos
Con el entorno virtual activo y estando en la carpeta del proyecto:
python manage.py startapp turnos
Esto genera la siguiente estructura:
turnos/ ├── __init__.py # Marca la carpeta como módulo Python ├── admin.py # Configuración del panel de administración ├── apps.py # Configuración de la app ├── migrations/ # Historial de cambios en la base de datos │ └── __init__.py ├── models.py # Definición de datos (Semana 4) ├── tests.py # Tests automatizados └── views.py # ← Acá vivimos hoy
Registrar la app en el proyecto
Django no sabe que la app existe hasta que la registramos. Abrir sistema_turnos/settings.py y agregar 'turnos' a INSTALLED_APPS:
# sistema_turnos/settings.py INSTALLED_APPS = [ 'django.contrib.admin', 'django.contrib.auth', 'django.contrib.contenttypes', 'django.contrib.sessions', 'django.contrib.messages', 'django.contrib.staticfiles', 'turnos', # ← Agregar esta línea ]
Si la app no está registrada acá, Django ignora sus modelos, URLs y migraciones. Es un error silencioso muy común.
Conceptos · 15 min
Flujo de una petición web en Django
Cuando alguien escribe localhost:8000/turnos/ en el navegador, ¿qué pasa exactamente dentro de Django? Entender este recorrido es entender cómo funciona cualquier framework web.
El recorrido completo
1
Navegador envía una petición HTTP
El ciudadano accede a localhost:8000/turnos/. El navegador genera un mensaje HTTP GET y lo envía al servidor Django.
2
Django busca en urls.py del proyecto
Django toma la URL /turnos/ y la compara con cada patrón definido en sistema_turnos/urls.py. Es como buscar en un directorio telefónico.
3
Encuentra una coincidencia y delega
El patrón turnos/ coincide. Django delega el resto al urls.py de la app turnos usando include().
4
La URL interna apunta a una vista
El urls.py de la app relaciona el patrón vacío "" con la función lista_turnos en views.py.
5
La vista procesa y responde
La función lista_turnos recibe el request, ejecuta la lógica necesaria (por ahora, sencilla) y devuelve una respuesta HTTP con el contenido HTML.
6
El navegador muestra la respuesta
Django envía el HTML al navegador. El ciudadano ve la página. Todo este recorrido ocurre en milisegundos.
Diagrama simplificado
Navegador ↓ GET /turnos/ sistema_turnos/urls.py → include('turnos.urls') ↓ turnos/urls.py → path("", views.lista_turnos) ↓ turnos/views.py → def lista_turnos(request): ↓ return HttpResponse("...") Navegador recibe HTML
Este flujo — URL → urls.py → views.py → respuesta — es el núcleo de Django. Todo lo que aprendemos en las próximas semanas (modelos, templates, formularios) se inserta en algún punto de este recorrido.
Código · 20 min
URLs y enrutamiento
El sistema de URLs de Django relaciona direcciones web con funciones Python. Es explícito: cada URL que el sistema acepta está declarada en algún urls.py.
El urls.py del proyecto — puerta de entrada
Este archivo ya existe, creado por Django. Lo modificamos para incluir las URLs de nuestra app:
# sistema_turnos/urls.py from django.contrib import admin from django.urls import path, include urlpatterns = [ path('admin/', admin.site.urls), path('turnos/', include('turnos.urls')), # ← Agregar ]
La función include() le dice a Django: "todo lo que empiece con turnos/, pasalo al archivo turnos/urls.py".
Crear urls.py dentro de la app
Este archivo no existe por defecto. Hay que crearlo en la carpeta turnos/:
# turnos/urls.py from django.urls import path from . import views urlpatterns = [ path('', views.lista_turnos, name='lista_turnos'), path('nuevo/', views.nuevo_turno, name='nuevo_turno'), ]
Anatomía de un path()
path( 'nuevo/', views.nuevo_turno, name='nuevo_turno' ) ↑ ↑ ↑ patrón URL función vista nombre interno
Patrón URL
La dirección relativa que Django debe reconocer. '' significa la raíz de este grupo (/turnos/). 'nuevo/' significa /turnos/nuevo/.
Función vista
La función Python en views.py que va a manejar esa URL. Se pasa sin paréntesis — Django la llama cuando la necesita.
Nombre interno
Un alias para referirnos a esta URL en templates y código sin escribir la ruta literal. Si la URL cambia, el alias sigue funcionando.
URLs con parámetros dinámicos
Para mostrar el detalle de un turno específico necesitamos pasar un ID en la URL:
# turnos/urls.py urlpatterns = [ path('', views.lista_turnos, name='lista_turnos'), path('nuevo/', views.nuevo_turno, name='nuevo_turno'), path('<int:turno_id>/', views.detalle_turno, name='detalle_turno'), ]
Con esto, /turnos/42/ llama a detalle_turno(request, turno_id=42). Django convierte el 42 de texto a entero automáticamente por el prefijo int:.
El trailing slash (/ al final) es convención en Django. APPEND_SLASH = True en settings redirige automáticamente si alguien omite el slash.
Código · 20 min
Vistas — el corazón del backend
Una vista es una función Python que recibe una petición HTTP y devuelve una respuesta HTTP. Es donde vive la lógica del sistema.
Vista mínima funcional
Toda vista sigue la misma firma: recibe request, devuelve una respuesta.
# turnos/views.py from django.http import HttpResponse def lista_turnos(request): return HttpResponse("<h1>Turnos disponibles</h1>")
En la Clase 11 vamos a reemplazar el HTML literal por templates. Por ahora, HttpResponse nos permite verificar que todo el flujo URL → vista → respuesta funciona.
El objeto request
Django entrega a cada vista toda la información sobre la petición entrante:
def lista_turnos(request): metodo = request.method # 'GET' o 'POST' usuario = request.user # Usuario autenticado parametros = request.GET # Parámetros en la URL datos_form = request.POST # Datos enviados por formulario return HttpResponse(f"Método: {metodo}")
Hoy usamos request.method. En las semanas siguientes vamos a explorar todos los demás atributos a medida que los necesitemos.
Vista con lógica condicional — GET vs POST
Esta es la estructura más común en Django. Una misma URL maneja dos comportamientos según el método HTTP:
def nuevo_turno(request): if request.method == 'POST': # El ciudadano envió el formulario nombre = request.POST.get('nombre', '') return HttpResponse( f"<p>Turno solicitado para: {nombre}</p>" ) # GET: mostrar el formulario vacío return HttpResponse(""" <form method="post"> <input name="nombre" placeholder="Tu nombre"> <button type="submit">Pedir turno</button> </form> """)
GET = "dame el formulario". POST = "procesá lo que completé". Esta distinción es fundamental en cualquier aplicación web.
Vista con parámetro de URL
def detalle_turno(request, turno_id): # turno_id llega como argumento porque lo declaramos # en urls.py con <int:turno_id> return HttpResponse( f"<h2>Detalle del turno #{turno_id}</h2>" f"<p>Fecha: martes 10 de junio, 10:30 hs</p>" )
Probar en el navegador: localhost:8000/turnos/1/, localhost:8000/turnos/99/. Django pasa el número como argumento automáticamente.
Práctica guiada · 25 min
Construir las rutas del sistema de turnos
Implementación completa paso a paso. Todos los archivos que hay que modificar o crear.
1
Crear la app y registrarla
# En la terminal: python manage.py startapp turnos
# sistema_turnos/settings.py — agregar 'turnos': INSTALLED_APPS = [ ... 'turnos', ]
2
Escribir las vistas
# turnos/views.py — reemplazar el contenido por: from django.http import HttpResponse def lista_turnos(request): turnos_disponibles = [ {"id": 1, "fecha": "Lunes 9 jun, 09:00", "tipo": "Jubilación"}, {"id": 2, "fecha": "Lunes 9 jun, 10:30", "tipo": "Asignación"}, {"id": 3, "fecha": "Martes 10 jun, 11:00", "tipo": "Consulta"}, ] items_html = "" for t in turnos_disponibles: items_html += f'<li>Turno #{t["id"]} — {t["fecha"]} ({t["tipo"]})</li>' html = f""" <h1>Sistema de Turnos — ANSES</h1> <h2>Turnos disponibles</h2> <ul>{items_html}</ul> <p><a href="/turnos/nuevo/">→ Solicitar nuevo turno</a></p> """ return HttpResponse(html) def nuevo_turno(request): if request.method == 'POST': nombre = request.POST.get('nombre', 'ciudadano') tipo = request.POST.get('tipo', 'consulta') return HttpResponse( f"<h2>✓ Turno registrado</h2>" f"<p>Nombre: {nombre}</p>" f"<p>Tipo: {tipo}</p>" f"<p><a href='/turnos/'>← Volver</a></p>" ) html = """ <h1>Solicitar turno</h1> <form method="post"> <p>Nombre: <input name="nombre" required></p> <p>Tipo: <select name="tipo"> <option>Jubilación</option> <option>Asignación</option> <option>Consulta</option> </select> </p> <button type="submit">Confirmar turno</button> </form> """ return HttpResponse(html) def detalle_turno(request, turno_id): return HttpResponse( f"<h2>Turno #{turno_id}</h2>" f"<p>Estado: Confirmado</p>" f"<p><a href='/turnos/'>← Volver al listado</a></p>" )
3
Crear el archivo de URLs de la app
Crear el archivo turnos/urls.py (no existe, hay que crearlo):
# turnos/urls.py from django.urls import path from . import views urlpatterns = [ path('', views.lista_turnos, name='lista_turnos'), path('nuevo/', views.nuevo_turno, name='nuevo_turno'), path('<int:turno_id>/', views.detalle_turno, name='detalle_turno'), ]
4
Conectar en el proyecto principal
# sistema_turnos/urls.py from django.contrib import admin from django.urls import path, include urlpatterns = [ path('admin/', admin.site.urls), path('turnos/', include('turnos.urls')), ]
5
Verificar en el navegador
python manage.py runserver
localhost:8000/turnos/ → Lista de turnos ✓ localhost:8000/turnos/nuevo/ → Formulario de solicitud ✓ localhost:8000/turnos/1/ → Detalle del turno 1 ✓ localhost:8000/turnos/99/ → Detalle del turno 99 ✓
Si alguna URL devuelve 404, el problema casi siempre está en el urls.py: un typo en el patrón, falta el include(), o la app no está en INSTALLED_APPS.

Referencia
Errores frecuentes en esta clase
Los problemas más comunes al trabajar con URLs y vistas por primera vez.
Page not found (404) — "Using the URLconf defined in sistema_turnos.urls, Django tried these URL patterns..."
La URL no coincide con ningún patrón definido. Revisar: (1) que el include('turnos.urls') esté en el urls.py del proyecto, (2) que el patrón en turnos/urls.py sea correcto, (3) que la URL en el navegador tenga el trailing slash.
ImportError: cannot import name 'lista_turnos' from 'turnos.views'
El nombre de la función en views.py no coincide exactamente con el que se importa en urls.py. Python distingue mayúsculas y minúsculas. Verificar que sea lista_turnos en ambos archivos.
ModuleNotFoundError: No module named 'turnos.urls'
El archivo turnos/urls.py no existe. Hay que crearlo manualmente — Django no lo genera con startapp.
CSRF verification failed. Request aborted.
Al enviar un formulario POST, Django exige un token de seguridad. En la Clase 16 vamos a ver esto en profundidad. Por ahora, para las pruebas, agregar {% csrf_token %} dentro del <form>, o en su defecto usar el decorador @csrf_exempt solo para desarrollo.
El servidor no refleja los cambios que hago en el código
El servidor de desarrollo de Django recarga automáticamente cuando detecta cambios. Si no ocurre, verificar que el archivo esté guardado (Ctrl+S / Cmd+S). En casos raros, detener el servidor (Ctrl+C) y reiniciarlo.
ValueError: The view turnos.views.lista_turnos didn't return an HttpResponse object
La función vista no tiene return o el return devuelve None. Toda vista debe devolver un objeto de respuesta. Verificar que cada rama del if/else tenga su propio return HttpResponse(...).
